# テスト駆動開発（TDD）

## 概要

まずテストを書く。失敗を確認する。最小限のコードを書いてテストを通す。

**核心原則：テストが失敗するところを見ていなければ、正しいものをテストしているか分からない。**

**ルールの形式的な側面より、ルールの本質が重要である。**

## 適用タイミング

**常に適用すること：**
- 新機能の実装
- バグ修正
- リファクタリング
- 動作変更

**例外（必ずユーザーに確認すること）：**
- 使い捨てのプロトタイプ
- 自動生成コード
- 設定ファイル

「今回だけ TDD をスキップしよう」と思ったら、立ち止まること。それは根拠のない言い訳である。

## 鉄則

```
失敗するテストなしに本番コードを書いてはならない
```

テストより先にコードを書いた場合は削除すること。最初からやり直す。

**例外なし：**
- 「参照用として残す」は禁止
- 「テストを書きながら適応させる」は禁止
- 「見ないようにする」は禁止
- 削除は削除を意味する

テストから実装を新たに始めること。以上。

## Red-Green-Refactor サイクル

### RED — 失敗するテストを書く

何が起きるべきかを示す最小限のテストを1つ書く。

**良い例：**
```typescript
import { describe, expect, it } from "vitest";
import { parseDrawioPath } from "./pathUtils";

describe("parseDrawioPath", () => {
  it("有効な.drawioファイルパスを受け取ったとき、パスをそのまま返す", () => {
    // 前提条件: 有効な.drawioファイルパス
    const result = parseDrawioPath("/tmp/diagram.drawio");

    // 検証: パスが返ること
    expect(result).toBe("/tmp/diagram.drawio");
  });
});
```
明確な名前、実際の動作をテスト、1つのことのみ検証。

**悪い例：**
```typescript
it("動作する", () => {
  const mock = vi.fn().mockReturnValue("/tmp/diagram.drawio");
  parseDrawioPath(mock as any);
  expect(mock).toHaveBeenCalled();
});
```
名前が曖昧、モックの動作をテストしている（実際のコードではない）。

**テストの要件：**
- 1つの動作のみ
- 動作を説明する明確な日本語名
- 実際のコードをテスト（避けられない場合のみモックを使用）
- `// 前提条件:` コメントでセットアップを説明
- `// 検証:` コメントでアサーションを説明
- テストデータには意味のある文字列を使用（例：`"/home/user/設計図.drawio"`, `"ネットワーク構成.drawio"`）

### RED の検証 — 失敗を確認する

**必須。絶対にスキップしないこと。**

```bash
pnpm test src/path/to/target.test.ts
```

以下を確認すること：
- テストが失敗する（エラーではない）
- 失敗メッセージが期待通りである
- 機能が存在しないことが原因で失敗している（タイポではない）

**テストが通過した場合：** 既存の動作をテストしている。テストを修正すること。

**テストがエラーになった場合：** エラーを修正し、正しく失敗するまで再実行すること。

### GREEN — 最小限のコード

テストを通過させる最もシンプルなコードを書く。

**良い例：**
```typescript
export function parseDrawioPath(filePath: string): string {
  if (!filePath.endsWith(".drawio")) {
    throw new Error(`Not a .drawio file: ${filePath}`);
  }
  return filePath;
}
```
テストを通過させるだけの最小限のコード。

**悪い例：**
```typescript
export function parseDrawioPath(
  filePath: string,
  options?: {
    validate?: boolean;
    normalize?: boolean;
    encoding?: BufferEncoding;
  }
): string {
  // YAGNI（今は必要ない）
}
```
過剰設計。テストが要求していないものを実装している。

テストを超えた機能の追加、他のコードのリファクタリング、「改善」は禁止。

### GREEN の検証 — 通過を確認する

**必須。**

```bash
pnpm test src/path/to/target.test.ts
```

以下を確認すること：
- テストが通過する
- 他のテストもすべて通過する（`pnpm test` で全体確認）
- 出力がクリーンである（エラーや警告がない）

**テストが失敗した場合：** テストではなくコードを修正すること。

**他のテストが失敗した場合：** 今すぐ修正すること。

### REFACTOR — クリーンアップ

GREEN になってから初めて行うこと：
- 重複の除去
- 名前の改善
- ヘルパーの抽出

テストが通過したままにすること。動作を追加しないこと。

### 繰り返し

次の機能に対して次の失敗するテストを書く。

## 良いテストの基準

| 品質 | 良い例 | 悪い例 |
|------|--------|--------|
| **最小限** | 1つのことのみ。名前に「かつ」が含まれる場合は分割する | `it("ファイル読み込みかつ検証かつ保存する")` |
| **明確** | 動作を説明する日本語名 | `it("テスト1")` |
| **意図を示す** | 期待する API をデモンストレーション | コードが何をすべきかを曖昧にしている |
| **コメント** | `// 前提条件:` と `// 検証:` で構造化 | コメントなし |
| **テストデータ** | 意味のある文字列（`"設計図.drawio"`, `"ネットワーク構成図.drawio"`） | `"foo"`, `"test1"` |

## なぜ順序が重要か

**「動作確認のためにテストを後から書けばよい」**

実装後に書いたテストはすぐに通過する。すぐに通過することは何も証明しない：
- 間違ったものをテストしているかもしれない
- 動作ではなく実装をテストしているかもしれない
- 忘れたエッジケースを見逃すかもしれない
- そのバグを発見したところを見ていない

テストファーストを徹底することで、テストの失敗を確認でき、本当に何かをテストしていることが証明される。

**「手動で全エッジケースをテスト済み」**

手動テストは場当たり的だ：
- 何をテストしたかの記録がない
- コード変更のたびに再実行できない
- プレッシャー下でケースを忘れやすい
- 「試したとき動いた」≠ 網羅的

自動テストは体系的。毎回同じ方法で実行される。

**「X 時間の作業を削除するのは無駄」**

埋没費用の誤謬だ。時間はすでに過ぎた。今の選択肢：
- 削除して TDD で書き直す（さらに X 時間、高い信頼性）
- そのまま後からテストを追加する（30分、低い信頼性、おそらくバグあり）

「無駄」とは信頼できないコードを保持することだ。本物のテストがない動作するコードは技術的負債である。

## よくある言い訳

| 言い訳 | 現実 |
|--------|------|
| 「単純すぎてテスト不要」 | 単純なコードも壊れる。テストに30秒かかる |
| 「後でテストする」 | すぐ通過するテストは何も証明しない |
| 「後から書いても同じ」 | 後から書くのは「これは何をするか？」。先に書くのは「何をすべきか？」 |
| 「手動テスト済み」 | 場当たり的≠体系的。記録がなく、再実行できない |
| 「X 時間削除は無駄」 | 埋没費用の誤謬。未検証のコードを保持することが無駄 |
| 「参照として残してテストを先に書く」 | それを適応させるだろう。それは後から書くことだ。削除とは削除を意味する |
| 「まず探索が必要」 | 構わない。探索を捨て、TDD で始めること |
| 「テストが難しい = 設計が不明確」 | テストを聞くこと。テストしにくい = 使いにくい |
| 「TDD は遅くなる」 | TDD はデバッグより速い。実用的 = テストファースト |

## 危険信号 — 立ち止まってやり直すこと

- テストより先にコードを書いた
- 実装後にテストを書いた
- テストがすぐに通過した
- テストがなぜ失敗したか説明できない
- テストを「後で」追加している
- 「今回だけ」と合理化している
- 「手動でテスト済み」と言っている
- 「後から書いても同じ目的を達成できる」と言っている
- 「精神を守れば儀式は不要」と言っている
- 「参照として残す」「既存コードを適応させる」と言っている
- 「X 時間かかったのに削除は無駄」と言っている
- 「TDD は独断的、私は実用的でいる」と言っている
- 「今回は違う、なぜなら...」と言っている

**これらはすべてを意味する：コードを削除して。TDD でやり直す。**

## このプロジェクト固有のパターン

### ファイル配置

- テストファイル名: `*.test.ts`（ソースと同じディレクトリに配置）

### Honoサーバーのテスト

HonoアプリはTestClientまたは `app.request()` を使ってHTTPレベルでテストする。

**RED の例：**
```typescript
import { describe, expect, it } from "vitest";
import { app } from "./server";

describe("GET /api/file", () => {
  it("存在するdrawioファイルのパスを渡したとき、ファイル内容をJSONで返す", async () => {
    // 前提条件: 実在するdrawioファイルパス（テスト用一時ファイル）
    const response = await app.request("/api/file?path=/tmp/設計図.drawio");

    // 検証: 200でファイル内容が返ること
    expect(response.status).toBe(200);
    const body = await response.json();
    expect(body.content).toBeDefined();
  });
});
```

### ファイルI/Oのテスト

実際のファイルを使うときは一時ディレクトリ（`mkdtemp`）を使う。`vi.mock`は避けられない場合のみ使用する。

**RED の例：**
```typescript
import { describe, expect, it } from "vitest";
import { mkdtemp, writeFile } from "node:fs/promises";
import { tmpdir } from "node:os";
import { join } from "node:path";
import { readDrawioFile } from "./fileUtils";

describe("readDrawioFile", () => {
  it("drawioファイルが存在するとき、ファイル内容を文字列で返す", async () => {
    // 前提条件: 一時ディレクトリに.drawioファイルを作成
    const tmpDir = await mkdtemp(join(tmpdir(), "cmux-drawio-test-"));
    const filePath = join(tmpDir, "ネットワーク構成図.drawio");
    await writeFile(filePath, "<mxGraphModel/>", "utf-8");

    // 検証: ファイル内容が返ること
    const result = await readDrawioFile(filePath);
    expect(result).toBe("<mxGraphModel/>");
  });
});
```

### テストコマンド

```bash
# 特定ファイルのみ
pnpm test src/path/to/target.test.ts

# 全テスト
pnpm test
```

## 例：バグ修正

**バグ：** 拡張子が `.drawio` 以外のファイルが受け入れられてしまう

**RED**
```typescript
it(".drawio以外の拡張子のパスは無効と判定される", () => {
  // 前提条件: .pngファイルのパス
  const result = isValidDrawioPath("/tmp/image.png");

  // 検証: 無効と判定されること
  expect(result).toBe(false);
});
```

**RED の検証**
```bash
$ pnpm test src/utils/pathValidator.test.ts
FAIL: expected false, received undefined
```

**GREEN**
```typescript
export function isValidDrawioPath(filePath: string): boolean {
  return filePath.endsWith(".drawio");
}
```

**GREEN の検証**
```bash
$ pnpm test src/utils/pathValidator.test.ts
PASS
```

**REFACTOR**
必要に応じてバリデーションロジックを整理する。

## 完了チェックリスト

作業完了とマークする前に確認すること：

- [ ] すべての新しい関数・メソッドにテストがある
- [ ] 各テストが失敗するところを実装前に確認した
- [ ] 各テストが期待通りの理由で失敗した（機能が存在しないから、タイポではない）
- [ ] 各テストを通過させる最小限のコードを書いた
- [ ] すべてのテストが通過する（`pnpm test`）
- [ ] 出力がクリーンである（エラーや警告がない）
- [ ] テストは実際のコードを使用している（避けられない場合のみモック）
- [ ] エッジケースとエラーがカバーされている

すべてのチェックボックスにチェックできない場合、TDD をスキップしている。やり直すこと。

## 行き詰まったとき

| 問題 | 解決策 |
|------|--------|
| テスト方法が分からない | 望ましい API を書く。まずアサーションを書く。ユーザーに確認する |
| テストが複雑すぎる | 設計が複雑すぎる。インターフェースを単純化する |
| すべてをモックしなければならない | コードが密結合すぎる。依存性注入を使う |
| テストのセットアップが膨大 | ヘルパーを抽出する。それでも複雑なら設計を単純化する |

## テストアンチパターン

モックやテストユーティリティを追加するときは、@testing-anti-patterns.md を読んで一般的な落とし穴を避けること：
- モックの動作をテストしている（実際の動作ではなく）
- テスト専用メソッドを本番クラスに追加している
- 依存関係を理解せずにモックしている

## 最終ルール

```
本番コード → テストが存在し、先に失敗していた
そうでなければ → TDD ではない
```

ユーザーの許可なしに例外なし。
